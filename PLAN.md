# Bangumi Show 应用计划

更新时间：2026-07-06

> 本文件只做产品与技术规划，不包含实现代码。当前仓库是空仓库，`PLAN.md` 是第一份项目文档。

## 1. 项目定位

Bangumi Show 是一个面向中文用户的日本动画查询应用，目标是让用户快速搜索、筛选、对比并查看动画作品信息。

核心体验应是“打开即查”，第一屏就是可用的查询界面，而不是宣传型落地页。

当前定位补充：

- 使用场景：本地或小范围自用，不以公开商业发布为目标。
- 内容范围：只覆盖日本动画；漫画、轻小说、游戏、国产动画、欧美动画暂不进入首版范围。
- 项目名称：保留 `Bangumi Show`。
- 核心入口：当前季度番表与搜索同等重要，首页默认态应优先展示当前季度内容。
- 后续形态：可能封装为 Electron 桌面应用；MVP 仍先按 Web 应用实现，但架构要预留桌面壳适配口。

## 2. 目标用户

- 想快速确认番剧中文名、日文名、放送季度、集数、评分和简介的普通观众。
- 追新番、补旧番、整理观看清单的动画爱好者。
- 需要查找声优、制作公司、关联作品、标签和外部链接的进阶用户。

## 3. MVP 范围

第一版只做查询与浏览，不做复杂社区功能。

必须包含：

- 关键词搜索：支持中文名、日文名、罗马音、别名。
- 搜索结果列表：展示封面、标题、年份/季度、类型、集数、评分、简短简介、数据来源。
- 作品详情页或详情面板：展示完整标题、简介、放送信息、制作信息、角色/声优、标签、关联作品、外部链接。
- 筛选与排序：按年份、季度、类型、状态、评分、标签筛选；按相关度、评分、开播时间排序。
- 最近搜索：本地保存最近搜索关键词，方便重复查询。
- 收藏/想看：MVP 可先做浏览器本地收藏，不引入账号系统。
- 响应式界面：桌面端适合快速扫描，移动端适合单手查询。

MVP 验收标准：

- 用户打开首页后，无需输入关键词也能看到当前季度番表或近期热门作品。
- 用户可以通过关键词搜索到作品，并看见匹配标题、放送年份/季度、类型、评分、简介和来源。
- 用户可以在搜索结果页打开详情面板，不离开当前查询上下文查看作品详情。
- 用户可以按年份、季度、类型、状态、评分、标签筛选，并按相关度、评分、开播时间排序。
- 用户可以把作品标记为想看、在看、看过、暂停或抛弃；收藏状态保存在浏览器本地。
- 用户刷新页面后，最近搜索和收藏状态仍然存在。
- 桌面端结果列表适合快速扫描；移动端筛选、排序和详情面板不遮挡核心操作。

暂不包含：

- 用户注册与跨设备同步。
- 评论、评分、社区动态。
- 视频播放。
- 爬虫采集自建全量数据库。
- 漫画、游戏、小说、国产动画、欧美动画等非日本动画条目，除非我们后续决定扩展。

## 4. 数据源策略

优先采用公开 API，通过服务端适配层统一字段，避免前端直接绑定某一个第三方数据结构。

推荐数据源：

- Bangumi API：作为首选数据源。优势是中文用户友好、别名和番组社区数据更贴近目标用户。
- AniList GraphQL API：作为补充数据源。优势是国际化数据、角色、制作人员、放送信息较完整。
- Jikan / MyAnimeList：只作为备选或补充。Jikan 是非官方 MAL API，不建议作为核心唯一依赖。

数据源原则：

- 每条数据保留来源标识和原始来源链接。
- 服务端负责请求第三方 API、缓存、限流、错误兜底。
- 前端只消费内部统一 API，不直接调用 Bangumi、AniList 或 Jikan。
- 对第三方条款、速率限制、授权方式保持可替换设计。
- Bangumi 作为主数据源；多来源字段冲突时，默认 Bangumi 优先，AniList 只补充 Bangumi 缺失或明显不足的字段。
- 因首版是本地小范围自用，缓存策略可以先偏简单，但仍要把缓存层封装出来，避免后续替换成本过高。
- 不建立自有全量数据库；只缓存查询结果、详情结果、图片缩略信息和必要的来源元数据。

多来源合并规则：

- 标题、中文别名、评分、标签、来源页面：优先使用 Bangumi。
- 角色、声优、制作人员、放送信息：Bangumi 优先；Bangumi 缺失时用 AniList 补充。
- 图片：优先使用 Bangumi 可用封面；如果来源限制明显，再考虑 AniList 图片或服务端缩略缓存。
- 排名、评分人数等统计字段不跨来源混算，应展示对应来源。
- 详情页需要标明主要来源，并在字段缺失、来源降级或补充字段来自其他来源时给出轻量提示。

数据源约束记录：

- 维护一个内部文档或常量表，记录各数据源的官方文档链接、认证方式、速率限制、关键字段覆盖、缓存 TTL 和失败降级策略。
- 实现时不要把速率限制数值硬编码在业务逻辑里，应集中配置，方便跟随官方文档调整。

参考链接：

- Bangumi API: https://bangumi.github.io/api/
- Bangumi API server: https://github.com/bangumi/server
- AniList API docs: https://docs.anilist.co/
- AniList docs repository: https://github.com/AniList/docs
- Jikan: https://jikan.moe/
- Jikan API docs: https://docs.api.jikan.moe/

## 5. 信息架构

核心实体：

- AnimeWork：动画作品。
- AnimeTitle：标题与别名。
- AnimeSourceRecord：第三方来源记录。
- AnimeStaff：制作人员、公司、声优。
- AnimeCharacter：角色信息。
- AnimeRelation：前作、续作、外传、剧场版等关联。
- SearchQuery：搜索请求和筛选条件。
- UserLocalCollection：本地收藏、想看、已看状态。
- SeasonSchedule：季度番表入口与日期/季度筛选结果。

作品核心字段：

- 内部 ID。
- 来源 ID 列表。
- 日文标题、中文标题、英文标题、罗马音标题、别名。
- 类型：TV、Movie、OVA、ONA、Special、Music 等。
- 放送年份、季度、开播日期、完结日期。
- 状态：未播出、放送中、已完结。
- 集数与单集时长。
- 简介。
- 封面图、主视觉图。
- 评分、排名、评分人数。
- 标签、题材、受众分级。
- 制作公司、导演、脚本、音乐、原作等。
- 角色与声优。
- 关联作品。
- 外部链接。

本地收藏字段：

- 内部作品 ID。
- 主要来源与来源 ID。
- 收藏状态：想看、在看、看过、暂停、抛弃。
- 标记时间与更新时间。
- 用于离线展示的最小快照：标题、封面、年份/季度、类型。

## 6. 产品页面规划

### 6.1 搜索首页

第一屏组成：

- 顶部搜索栏，支持即时建议。
- 搜索范围切换：全部、TV、电影、OVA/ONA。
- 季度快捷入口：当前季度、上一季度、下一季度、年度热门。
- 结果区：默认展示当前季度番表，避免空白首页。
- 当前季度番表应作为核心入口，而不是次级快捷链接。

### 6.2 搜索结果页

桌面端建议为左侧筛选、右侧结果列表。

移动端建议筛选进入底部抽屉，结果卡片纵向排列。

结果卡片展示：

- 封面。
- 中文名与日文名。
- 年份、季度、类型、状态。
- 简介摘要。
- 评分与评分人数。
- 标签。
- 收藏/想看快捷操作。

### 6.3 作品详情

MVP 优先采用详情面板，桌面端从结果区右侧或覆盖层打开，移动端使用底部抽屉或全屏面板。暂不优先做独立详情路由。

详情面板结构：

- 顶部：封面、主标题、别名、评分、放送状态、收藏操作。
- 摘要：简介、开播日期、集数、类型、制作公司。
- 信息 Tab：概览、角色、制作人员、关联作品、外部链接。
- 底部：同季度作品、相似标签作品、前后关联作品。

### 6.4 本地收藏页

MVP 使用浏览器本地存储即可。

收藏状态：

- 想看。
- 在看。
- 看过。
- 暂停。
- 抛弃。

首版收藏功能只做状态标记，不做备注、个人评分、观看进度和跨设备同步。

后续如果做账号系统，再迁移到服务端用户表。

## 7. 技术架构建议

由于当前仓库为空，建议采用前后端同仓的 Web 应用结构，先保证开发效率。

推荐初始技术方案：

- 应用框架：Next.js + TypeScript。
- UI：React 组件 + Tailwind CSS；后续可引入成熟组件库。
- 服务端：Next.js Route Handlers 或独立 API 层。
- 数据访问：第三方 API adapter + 统一 normalized model。
- 缓存：开发期可先用内存或 SQLite；生产期再扩展 Redis / PostgreSQL。
- 搜索：MVP 先使用第三方 API 搜索；后续如果需要更快和更稳定的检索，再引入 Meilisearch 或自建索引。
- 测试：单元测试覆盖 adapter 和 normalizer；Playwright 覆盖核心查询流程。

Electron 预留原则：

- 保持核心业务逻辑、数据 adapter、normalizer、收藏状态管理与 UI 框架解耦，避免直接依赖浏览器全局对象。
- 本地存储不要在业务代码中直接散落调用 `localStorage`，应封装为 collection repository，未来可替换为 Electron 本地文件、SQLite 或 IndexedDB。
- 导入/导出、图片缓存、配置文件、日志等系统能力统一走平台 adapter，Web 版先提供浏览器实现，Electron 版后续提供桌面实现。
- 内部 API 设计保持 HTTP/函数边界清晰，避免前端直接访问第三方 API；Electron 版可以复用同一套 adapter 作为主进程或本地服务能力。
- UI 布局要考虑固定窗口、可调整窗口和桌面快捷键，但 MVP 不先绑定 Electron 菜单、托盘、自动更新等能力。
- 避免引入只能在纯浏览器部署或只能在 Node/Electron 环境运行的硬依赖；必须引入时放在平台层隔离。

本地小范围自用部署建议：

- 开发与自用阶段可优先使用本地 Next.js 服务。
- 如需在局域网内访问，可用同一台机器部署，配合简单环境变量管理 API 配置。
- 暂不需要账号系统、复杂权限、后台管理或生产级多租户能力。

可替代方案：

- 如果希望前后端边界更清晰，可采用 React/Vite 前端 + Node.js API 服务。
- 如果希望桌面体验优先，可后续封装为 Electron 桌面应用；Tauri 也可作为备选，但当前预留方向以 Electron 为主。

## 8. 内部 API 草案

这些是概念接口，用来帮助产品和前端对齐，不代表最终实现。

- `GET /api/search`：关键词搜索动画。
- `GET /api/anime/{id}`：获取作品详情。
- `GET /api/anime/{id}/relations`：获取关联作品。
- `GET /api/schedule`：按季度或日期获取放送表；首页核心入口。
- `GET /api/suggest`：搜索联想。
- `GET /api/sources/status`：检查第三方数据源健康状态。

参数草案：

- `GET /api/search?q=&type=&year=&season=&status=&tag=&sort=&page=&pageSize=`
- `GET /api/anime/{id}?include=characters,staff,relations,links`
- `GET /api/schedule?year=&season=&weekday=&type=`
- `GET /api/suggest?q=&limit=`

排序值草案：

- `relevance`：相关度。
- `score`：评分。
- `airDate`：开播时间。
- `popularity`：热度或评分人数，字段可用时再启用。

接口返回应统一包含：

- `data`：标准化结果。
- `source`：主要来源。
- `sourceLinks`：原始页面链接。
- `cachedAt`：缓存时间。
- `warnings`：数据缺失、来源异常、降级说明。

缓存建议：

- 搜索结果和季度番表可以设置较短 TTL。
- 已完结作品详情可以设置较长 TTL。
- 放送中作品详情和当前季度番表需要更频繁刷新。
- 来源异常时优先返回可用缓存，并在 `warnings` 中说明。

## 9. 视觉与交互方向

设计目标：信息密度高、扫描快、不要做成营销页。

界面原则：

- 搜索框始终突出，结果和筛选是主角。
- 封面图要清晰可辨，避免过度模糊或装饰性背景。
- 桌面端偏工具化，适合横向比较。
- 移动端减少层级，优先搜索、收藏、查看详情。
- 使用明确的图标按钮承载收藏、筛选、排序、返回等操作。
- 保持多色但克制的视觉系统，不让页面被单一色系淹没。
- 当前季度番表需要有清晰的日期、星期或放送状态线索，方便用户快速扫番。
- 详情面板打开后要保留搜索结果上下文，关闭后回到原来的滚动位置和筛选条件。

## 11. 风险与应对

- 第三方 API 限流或变更：通过服务端 adapter、缓存和降级数据源隔离风险。
- 数据字段不一致：使用统一 normalizer，并保留原始 source record 便于排查。
- 中文名和别名匹配不稳定：搜索结果需要展示匹配原因，后续可加入本地别名索引。
- 图片热链限制：优先尊重来源策略，必要时使用服务端代理或缓存缩略图。
- 用户收藏数据迁移：MVP 本地收藏要预留导出格式，方便未来账号系统导入。
- 法务与授权：保留来源链接和数据来源说明，不抓取或再分发超出许可的数据。
- 当前季度数据变化频繁：对当前季度番表使用更短缓存，并在 UI 中展示更新时间。
- 小范围自用导致测试容易被忽略：至少保留 adapter、normalizer 和核心查询流程测试，避免 API 变更时无感损坏。
- 数据源健康不可见：提供 `/api/sources/status`，记录最近错误、限流、缓存命中和降级情况。
- 后续 Electron 封装改动过大：提前隔离平台能力和本地存储层，避免业务代码直接绑定 `localStorage`、文件系统或 Electron API。

## 12. 待确认问题

需要我们一起决定：

- 是否优先做中文体验，还是中/日/英三语都作为一等公民？
